# Module 4: Events & Triggers

## Overview
A workflow run is triggered by a specific event. These events can originate from activity occurring natively in your GitHub repository, occur on a strict time schedule, or be triggered manually or via an external webhook.

**Official Reference:** [Events that trigger workflows](https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows)

---

## Core Trigger Categories

### 1. Webhook Events
Workflows can be triggered by webhook events generated by GitHub when certain activities happen in your repository.
- **`push`**: Runs your workflow when you push a commit or push a tag.
- **`pull_request`**: Runs your workflow when activity occurs on a pull request (e.g., when it is opened, synchronized, or reopened).
- **`issues`**: Runs when an issue is created, modified, assigned or closed.
- **`release`**: Runs when a GitHub Release is published.

### 2. Scheduled Events
The `schedule` event allows you to trigger a workflow at a scheduled time using POSIX cron syntax.
```yaml
on:
  schedule:
    # Runs at 00:00 every day
    - cron: '0 0 * * *'
```

### 3. Manual Triggers
- **`workflow_dispatch`**: Allows you to manually trigger a workflow run from the GitHub API, GitHub CLI, or directly from the Actions tab in the GitHub UI. This is incredibly powerful because you can also configure **custom inputs** to be presented to the user clicking the button!

---

## Advanced Trigger Syntax

### Using Filters (Paths, Branches, Tags)
You don't always want a workflow to run on *every* push to the repository. You can pass filters to make your triggers highly specific.

```yaml
on:
  push:
    branches:
      - main
      - 'releases/**'
    paths:
      - 'src/**'        # Only run if code inside src/ changed
    paths-ignore:
      - 'docs/**'       # Do not run if ONLY documentation changed
```

### Using Activity Types
Some events have multiple types of activity that trigger them. For example, the `pull_request` event triggers by default when a PR is opened, reopened, or synchronized. You can pin it to specific activity types using `types`.

```yaml
on:
  pull_request:
    types: [opened, labeled, closed]
```

## Example Workflow: Trigger Deep Dive

You can find the standalone code for this in `triggers-demo.yml`.

```yaml
# .github/workflows/triggers-demo.yml

name: Advanced Events and Triggers

# A comprehensive trigger block
on:
  # 1. Trigger when changes are pushed to main, but ONLY if files inside 'src/' change.
  push:
    branches:
      - main
    paths:
      - 'src/**'
      
  # 2. Trigger on pull requests targeting the main branch when they are successfully closed (merged).
  pull_request:
    branches:
      - main
    types: 
      - closed
      
  # 3. Scheduled Trigger: Run at 2:00 AM UTC every Sunday
  schedule:
    - cron: '0 2 * * 0'
    
  # 4. Manual Trigger with custom UI inputs!
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy to'
        required: true
        default: 'staging'
        type: choice
        options:
        - staging
        - production
      reason:
        description: 'Reason for manual trigger'
        required: false
        type: string

jobs:
  demonstrate-triggers:
    runs-on: ubuntu-latest
    steps:
      - name: Print the Event Name
        run: echo "Workflow triggered by: ${{ github.event_name }}"

      - name: Print Manual Input Options (If applicable)
        # We can use the 'if' conditional to only run this step if the trigger was manual
        if: github.event_name == 'workflow_dispatch'
        run: |
          echo "Deploying to Environment: ${{ inputs.environment }}"
          echo "Reason: ${{ inputs.reason }}"
          
      - name: Action on Pull Request Close
        if: github.event_name == 'pull_request' && github.event.action == 'closed'
        run: echo "A pull request was recently closed!"
        
      - name: Scheduled Routine
        if: github.event_name == 'schedule'
        run: echo "Running our scheduled weekend task!"
```

---

## How to Execute It

1. Place the `triggers-demo.yml` file into `.github/workflows/`.
2. Commit and distribute it to your repository.
3. **To test the manual trigger via UI**: Go to the Actions tab, select "Advanced Events and Triggers", click "Run workflow" on the right. You will be prompted with a beautiful dropdown menu to select the "Environment" and a text box to input the "Reason".
4. **To see the PR trigger**: Check out a new branch, make a change, open a Pull Request against `main`, and then merge it. The pipeline will ONLY execute on the close activity.
